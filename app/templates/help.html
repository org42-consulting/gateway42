Overview

Gateway42 is a secure, multi-user proxy gateway for Ollama. It sits between your users and a shared Ollama instance, providing:

  • OpenAI-compatible API — clients built for the OpenAI API work with Gateway42 without modification.
  • API key authentication — every user receives a unique key; only active users can send requests.
  • Per-user rate limiting — configurable maximum requests per minute per user.
  • Audit logging — every prompt and response is stored with a timestamp, user, and model name.
  • Admin dashboard — manage users, configure Ollama, download or delete models, and review logs.

Dashboard

The Dashboard is the main landing page after login. It shows two panels:

Ollama Status

Displays whether the configured Ollama instance is reachable, its base URL and port, and the list of currently installed models. This panel is refreshed every time the page loads. If Ollama is unreachable, the status shows INACTIVE and the model list is unavailable.

Registered Users

A table of every user account, with the following columns and actions:

Column / ActionDescription
EmailThe user's registered email address. Used as the username.
Status Clicking the status badge toggles the user between ACTIVE and DISABLED. Only active users can make API requests. Newly registered users start as DISABLED.
Rate / min The maximum number of API requests this user can make per minute. Edit the number inline and click Update to save. Valid range: 1–1000.
New API Key Generates a fresh API key for the user and immediately invalidates the old one. The new key is displayed once — copy it before closing the page.
Export CSV Downloads all audit log entries for this user as a CSV file. This export is also required before the user can be deleted.
Delete Permanently removes the user and all their log entries. You must export the user's CSV first. A confirmation page is shown before the deletion is committed.

Register New User

Enter an email address and click Register User. A unique API key is generated automatically and displayed on the next page. The user starts in DISABLED status — activate them manually once you have shared their API key securely.

Reset System

Permanently deletes all audit logs and all rate-limit counters. User accounts and application settings are not affected. A confirmation dialog is shown first. This action cannot be undone.

Settings

Ollama Settings

Configure the base URL and port of your Ollama instance. The gateway connects to {base_url}:{port}/api/chat for completions and /api/tags for model discovery.

  • Ollama Base URL — scheme and hostname, e.g. http://127.0.0.1.
  • Port Number — typically 11434 (Ollama default). Valid range: 1–65535.
  • Test Connectivity — sends a quick probe to Ollama and reports the version number.
  • Save Settings — persists the URL and port to the database, overriding environment defaults.

Model Management

Displays all models currently installed on the Ollama instance, including their disk size. This panel is greyed out when Ollama is unreachable.

  • Delete — removes a model from the Ollama instance. A confirmation dialog is shown first. This frees disk space but does not affect audit logs that referenced that model.
  • Download Model — enter a model name (e.g. llama3.2:latest) and click Pull. A live progress bar tracks the download layer by layer, showing bytes transferred and overall percentage. Model names are available at ollama.com/models.

Change Admin Password

Updates the admin account password. Requirements: at least 8 characters, one uppercase letter, one lowercase letter, and one digit. You must provide the current password to confirm the change.

Register New User

Same as the Register New User panel on the Dashboard — provided here for convenience.

Reset System

Same as the Reset System action on the Dashboard — see above for details.

Logs

Every API request processed by Gateway42 is recorded. The Logs page shows the most recent 200 entries by default.

The auto-refresh dropdown polls the server in the background and updates the log table in place without a full page reload. Options: Off, 5 s, 10 s (default), 30 s, 60 s.

Downloads all log entries (not just the current 200) as a CSV file with columns: log_id, email, model, prompt, response, timestamp.

Same as the Reset System action on the Dashboard — clears all logs. Requires confirmation.

API Usage

Gateway42 exposes an OpenAI-compatible API. Any client or library that targets the OpenAI API can be pointed at Gateway42 by changing the base URL and providing a Gateway42 API key.

Base URL

Replace the OpenAI base URL with your Gateway42 host:

http://<your-host>:7000/v1

Authentication

Pass the user's API key as a Bearer token in the Authorization header:

Authorization: Bearer <api_key>

Requests with a missing, invalid, or deactivated key are rejected with 401 Unauthorized.

Endpoints

EndpointMethodDescription
/v1/models GET Returns the list of models installed on Ollama in OpenAI format.
/v1/chat/completions POST Chat completion endpoint. Accepts OpenAI-format request bodies. Set "stream": true for Server-Sent Events streaming.
/health GET Returns {"status": "ok"}. No authentication required. Useful for uptime monitoring.

Example — cURL

curl http://<host>:7000/v1/chat/completions \
  -H "Authorization: Bearer <api_key>" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "llama3.2:latest",
    "messages": [{"role": "user", "content": "Hello!"}]
  }'

Example — Python (openai SDK)

from openai import OpenAI

client = OpenAI(
    base_url="http://<host>:7000/v1",
    api_key="<api_key>",
)

response = client.chat.completions.create(
    model="llama3.2:latest",
    messages=[{"role": "user", "content": "Hello!"}],
)
print(response.choices[0].message.content)

Rate Limiting

Gateway42 enforces a sliding window rate limit per user. The window is 60 seconds. If a user sends more requests than their configured limit within any 60-second window, the gateway rejects further requests with 429 Too Many Requests until the window clears.

  • The default rate limit for new users is set by the DEFAULT_RATE_LIMIT environment variable (default: 10 req/min).
  • Each user's limit can be adjusted individually from the Dashboard.
  • Rate-limit counters are reset by the Reset System action or when a user is deleted.

Running with Docker

Gateway42 ships with a Dockerfile and a docker-compose.yml. It assumes Ollama is already running on the host machine at localhost:11434 (the default Ollama port). The container reaches the host via host.docker.internal.

Prerequisites

  • Docker 24 or later.
  • Docker Compose v2 (bundled with Docker Desktop).
  • Ollama installed and running on the host: ollama.com/download.

How host networking works

Inside a Docker container, localhost refers to the container itself — not the machine running Docker. Gateway42 uses host.docker.internal instead, which Docker resolves to the host's IP address:

  • macOS / Windows — Docker Desktop provides host.docker.internal automatically.
  • Linux — the extra_hosts: host.docker.internal:host-gateway entry in docker-compose.yml adds the same mapping. No extra setup needed.

Quick start

# 1. Make sure Ollama is running on the host
ollama serve   # or start it via the Ollama app

# 2. Edit docker-compose.yml — set your secrets:
#    GW42_SECRET_KEY — any long random string
#    ADMIN_PASSWORD            — your chosen admin password

# 3. Build and start
docker compose up -d

# 4. Open the admin UI
open http://localhost:7000

Using a published image

To run Gateway42 without cloning the repository or building locally, replace the build: . line in docker-compose.yml with an image: reference: 

services:
  gateway42:
    image: ghcr.io/YOUR_ORG/gateway42:latest   # replace with the published image tag
    # build: .  ← remove or comment out this line

Store your secrets in a .env file in the same directory:

# .env
GW42_SECRET_KEY=your-long-random-secret
ADMIN_PASSWORD=your-admin-password

Reference them in docker-compose.yml:

    environment:
      GW42_SECRET_KEY: "${GW42_SECRET_KEY}"
      ADMIN_PASSWORD: "${ADMIN_PASSWORD}"

Then pull and start:

docker compose pull
docker compose up -d

Persistent data

Two named Docker volumes are created automatically:

  • gateway42-db — SQLite database (users, logs, settings).
  • gateway42-logs — application log files.

Data survives container restarts and upgrades. To wipe everything: docker compose down -v.

Run without Compose

docker build -t gateway42:latest .

docker run -d \
  -p 7000:7000 \
  --add-host=host.docker.internal:host-gateway \
  -e GW42_SECRET_KEY="your-secret" \
  -e ADMIN_PASSWORD="your-password" \
  -e OLLAMA_URL="http://host.docker.internal:11434/api/chat" \
  -v gateway42-db:/gateway/db \
  -v gateway42-logs:/gateway/logs \
  --name gateway42 \
  gateway42:latest

The --add-host flag is required on Linux. On macOS and Windows with Docker Desktop it is optional but harmless.

Upgrading

If you built the image locally:

docker compose build    # rebuild the gateway42 image
docker compose up -d    # replace the running container

If you are using a published image:

docker compose pull     # fetch the latest image
docker compose up -d    # replace the running container

The SQLite database schema is migrated automatically on startup — no manual steps required.

Environment variables

Set variables directly in docker-compose.yml, or use a .env file in the project root and reference them with ${VARIABLE_NAME}: 

# .env
GW42_SECRET_KEY=your-long-random-string
ADMIN_PASSWORD=your-admin-password
ADMIN_EMAIL=admin@example.com

Running on Kubernetes

Ready-to-use manifests are provided in the k8s/ directory. They deploy Gateway42 as a single-replica Deployment backed by PersistentVolumeClaims for the database and logs.

Files

FilePurpose
k8s/secret.yamlStores GW42_SECRET_KEY and ADMIN_PASSWORD as a Kubernetes Secret.
k8s/configmap.yamlNon-sensitive configuration (Ollama URL, rate limits, log level, etc.).
k8s/pvc.yamlPersistentVolumeClaims for the database (1 Gi) and logs (2 Gi).
k8s/deployment.yamlThe Deployment definition with health probes and resource limits.
k8s/service.yamlClusterIP Service on port 80 and an Ingress resource for external access.

Deploy

# 1. Build and push the image to your registry
docker build -t your-registry/gateway42:latest .
docker push your-registry/gateway42:latest

# 2. Update the image reference in k8s/deployment.yaml
#    image: your-registry/gateway42:latest

# 3. Set your secrets (edit k8s/secret.yaml before applying,
#    or use kubectl create secret directly)
kubectl create secret generic gateway42-secret \
  --from-literal=GW42_SECRET_KEY="your-long-random-string" \
  --from-literal=ADMIN_PASSWORD="your-admin-password"

# 4. Apply all manifests
kubectl apply -f k8s/configmap.yaml
kubectl apply -f k8s/pvc.yaml
kubectl apply -f k8s/deployment.yaml
kubectl apply -f k8s/service.yaml

# 5. Check rollout status
kubectl rollout status deployment/gateway42

Access the UI

The Service is a ClusterIP on port 80. To access it from outside the cluster, either:

  • Update k8s/service.yaml to use type: LoadBalancer, or
  • Configure the Ingress in k8s/service.yaml with your domain and an ingress controller (e.g. nginx-ingress), or
  • Use port-forwarding for local access: kubectl port-forward svc/gateway42 7000:80

Connecting to Ollama

The default OLLAMA_URL in k8s/configmap.yaml points to http://ollama:11434/api/chat, which assumes Ollama is deployed as a Service named ollama in the same namespace. Alternatives:

  • Separate namespace: use the full DNS name http://ollama.<namespace>.svc.cluster.local:11434/api/chat.
  • External Ollama: set OLLAMA_URL to the external IP or hostname.
  • Runtime override: change the URL any time from the Settings page — it is saved to the database and takes effect immediately.

Scaling

Gateway42 uses a SQLite database, which supports only one writer at a time. Running more than one replica is therefore not recommended without replacing the database backend. For high-throughput deployments, keep replicas: 1 and scale horizontally at the Ollama level instead.

Health checks

The Deployment configures both a readiness probe and a liveness probe against GET /health. Kubernetes will not route traffic to the pod until the readiness probe succeeds, and will restart the pod if the liveness probe fails.
